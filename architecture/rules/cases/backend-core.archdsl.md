# Architecture Rule DSL — LLM Generation Case

## Task

Generate a valid `backend-core.archdsl.yaml` for the Routa project following the schema below.

## Schema

```yaml
version: "1"
name: <string>
description: <string>
rules:
  - id: <kebab-case>
    title: <human readable>
    suite: boundaries | cycles
    source:
      folder: <relative path>
    constraint:
      type: must_not_depend_on | no_cycles
      target:             # only for must_not_depend_on
        folder: <relative path>
```

## Constraints

- `version` must be the string `"1"`.
- `id` must be unique within the file and use `kebab-case`.
- `suite` is `boundaries` for dependency direction rules, `cycles` for cycle detection.
- `constraint.type` is `must_not_depend_on` for boundary rules; `no_cycles` for cycle rules.
- `target` is required when `constraint.type` is `must_not_depend_on`; omit otherwise.
- No markdown fences in the output YAML.
- Preserve exact field names — do not add extra fields.

## Context: Routa Crate Layout

```
crates/
  routa-core/src/         — domain model, database, state
  routa-server/src/       — Axum HTTP layer, depends on routa-core
  routa-cli/src/          — CLI commands, depends on routa-core + routa-server
  routa-scanner/src/      — static analysis helpers, minimal deps
  routa-entrix/src/       — fitness evaluation engine
src/
  core/                   — Next.js TypeScript core logic
  app/                    — Next.js app router pages and API routes
```

## Example Output

```yaml
version: "1"
name: "backend-core"
description: "Core backend architecture rules for Routa."
rules:
  - id: core-no-server-imports
    title: "routa-core must not import from routa-server"
    suite: boundaries
    source:
      folder: "crates/routa-core/src"
    constraint:
      type: must_not_depend_on
      target:
        folder: "crates/routa-server/src"
  - id: no-cycles-src-core
    title: "No circular dependencies in src/core TypeScript modules"
    suite: cycles
    source:
      folder: "src/core"
    constraint:
      type: no_cycles
```

## Validation Contract

A rule file generated by an LLM is considered valid when:
1. `yaml.parse()` succeeds without errors.
2. `version === "1"`.
3. Every rule has `id`, `title`, `suite`, `source.folder`, `constraint.type`.
4. For `must_not_depend_on` rules, `constraint.target.folder` is present.
5. All `id` values are unique within the file.
